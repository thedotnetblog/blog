يمر MCP (بروتوكول سياق النموذج) بلحظته المميزة. إذا كنت تتابع منظومة وكلاء الذكاء الاصطناعي، فلاحظت على الأرجح انتشار خوادم MCP في كل مكان — حيث تمنح الوكلاء القدرة على التفاعل مع الأدوات والخدمات الخارجية من خلال بروتوكول موحد.

الآن [أصبح خادم Azure DevOps MCP متاحاً في Microsoft Foundry](https://devblogs.microsoft.com/devops/remote-mcp-server-preview-in-microsoft-foundry/)، وهذه من التكاملات التي تجعلك تفكر في الإمكانات العملية.

## ما الذي يحدث فعلاً

أصدرت Microsoft بالفعل خادم Azure DevOps MCP كـ[معاينة عامة](https://devblogs.microsoft.com/devops/azure-devops-remote-mcp-server-public-preview) — ذلك هو الخادم نفسه. الجديد هو التكامل مع Foundry. يمكنك الآن إضافة خادم Azure DevOps MCP إلى وكلاء Foundry مباشرةً من كتالوج الأدوات.

لمن لم يتعرف على Foundry بعد: إنها المنصة الموحدة من Microsoft لبناء وإدارة التطبيقات والوكلاء المدعومة بالذكاء الاصطناعي على نطاق واسع. الوصول إلى النماذج، والتنسيق، والتقييم، والنشر — كل ذلك في مكان واحد.

## كيفية الإعداد

الإعداد مباشر بشكل مفاجئ:

1. في وكيل Foundry الخاص بك، انتقل إلى **Add Tools** > **Catalog**
2. ابحث عن "Azure DevOps"
3. اختر Azure DevOps MCP Server (preview) وانقر **Create**
4. أدخل اسم مؤسستك وتواصل

هذا كل شيء. الوكيل لديك الآن وصول إلى أدوات Azure DevOps.

## التحكم في ما يمكن للوكيل الوصول إليه

هذا هو الجانب الذي أقدره: لست مقيداً بنهج "الكل أو لا شيء". يمكنك تحديد الأدوات المتاحة لوكيلك. لذا إذا أردته فقط قراءة عناصر العمل دون لمس المسارات، يمكنك ضبط ذلك. مبدأ الصلاحية الأدنى، مطبقاً على وكلاء الذكاء الاصطناعي لديك.

هذا مهم في سيناريوهات المؤسسات حيث لا تريد أن يُشغّل الوكيل مسار نشر بالخطأ لأن أحدهم طلب منه "المساعدة في الإصدار".

## لماذا هذا مثير للاهتمام لفرق .NET

فكر في ما يتيحه هذا عملياً:

- **مساعدو تخطيط السبرينت** — وكلاء يسحبون عناصر العمل ويحللون بيانات السرعة ويقترحون سعة السبرينت
- **بوتات مراجعة الكود** — وكلاء يفهمون سياق طلب السحب الخاص بك لأنهم يستطيعون فعلاً قراءة المستودعات وعناصر العمل المرتبطة
- **الاستجابة للحوادث** — وكلاء يمكنهم إنشاء عناصر عمل والاستعلام عن النشرات الأخيرة وربط الأخطاء بالتغييرات الأخيرة
- **تأهيل المطورين** — "ما الذي يجب أن أعمل عليه؟" تحصل على إجابة حقيقية مستندة إلى بيانات المشروع الفعلية

بالنسبة لفرق .NET التي تستخدم بالفعل Azure DevOps لمسارات CI/CD وإدارة المشاريع، فإن امتلاك وكيل ذكاء اصطناعي يمكنه التفاعل مع تلك الأنظمة مباشرةً خطوة مهمة نحو أتمتة مفيدة حقيقية (وليس مجرد روبوت محادثة كخدمة).

## الصورة الأكبر لـ MCP

هذا جزء من اتجاه أوسع: تصبح خوادم MCP الطريقة المعيارية التي تتفاعل بها وكلاء الذكاء الاصطناعي مع العالم الخارجي. نراها لـ GitHub وAzure DevOps وقواعد البيانات وواجهات برمجية SaaS — وتصبح Foundry المحور الذي تجتمع فيه هذه الاتصالات.

إذا كنت تبني وكلاء في منظومة .NET، فإن MCP يستحق الاهتمام. البروتوكول موحد، والأدوات تنضج، وتكامل Foundry يجعله في متناول يدك دون الحاجة إلى توصيل خوادم يدوياً.

## خلاصة القول

خادم Azure DevOps MCP في Foundry في مرحلة المعاينة، لذا توقع أن يتطور. لكن سير العمل الأساسي متين: اتصل، وضبط وصول الأدوات، واترك وكلاءك يعملون مع بيانات DevOps الخاصة بك. إذا كنت بالفعل في منظومة Foundry، فهذا على بُعد بضع نقرات. جربه وانظر إلى سير العمل التي يمكنك بناؤها.

اطلع على [الإعلان الكامل](https://devblogs.microsoft.com/devops/remote-mcp-server-preview-in-microsoft-foundry/) للحصول على خطوات الإعداد التفصيلية والمزيد من المعلومات.
